# 챕터 7. 변경 관리 (2차 견적)

> 이 챕터를 읽고 나면 — 클라이언트의 변경 요청을 받았을 때 1차 견적을 보존하면서 2차 견적을 만들고, 마감재 변경 이력을 추적하며, 분쟁이 발생했을 때 증빙으로 활용할 수 있게 됩니다.

이 챕터는 **분쟁 예방의 핵심**입니다. 인테리어 업계의 분쟁 70%가 "이 가격 합의 안 했다"에서 시작됩니다.

---

## 7.1 ★ 변경 = 견적 2차 (절대 원칙)

Suplex의 가장 중요한 운영 원칙:

> **클라이언트가 변경을 요청하면, 1차 견적을 수정하지 마세요. 새로운 2차 견적을 만드세요.**

### 왜 그래야 하나

1. **분쟁 시 증거가 망가집니다** — 1차를 수정하면 클라이언트가 "원래 견적이 어땠는지" 주장하기 시작합니다.
2. **변경 사항 추적이 어려워집니다** — 어떤 부분이 어떻게 바뀌었는지 시간 따라 모릅니다.
3. **클라이언트 신뢰가 깨집니다** — 같은 견적서가 매번 달라지면 신뢰가 가지 않습니다.
4. **회계·세무 연결이 흐트러집니다** — 발행 차수별로 명확해야 정산이 깔끔합니다.

### 어떻게 다르게 하나

```
❌ 1차 견적의 강마루 단가 75,000 → 78,000으로 직접 수정
   (잘못된 방식)

✅ 1차 견적은 그대로 두고, 새로운 2차 견적을 만들어 78,000 입력
   (Suplex 표준 방식)
```

---

## 7.2 1차 → 2차 흐름

### 단계별

```
[클라이언트 변경 요청 도착]
        ↓
[마감재 탭에서 변경 항목 상태 = CHANGED로]
        ↓
[1차 견적 상태 = SUPERSEDED로 잠그기]
        ↓
[새 견적 생성 (또는 1차 복제)]
        ↓
[변경된 라인만 수정]
        ↓
[직전 차수와 비교 — AI 자동 요약 → footerNotes]
        ↓
[2차 PDF 출력 → 클라이언트에게 발송]
```

자세한 실전 흐름은 [Playbook #1](playbooks/01-30py-apartment-full.md) 단계 ⑤~⑨ 참고.

### 견적 상태 흐름 (변경 시)

| 시점 | 1차 상태 | 2차 상태 |
|---|---|---|
| 1차 발송 직후 | SENT | (없음) |
| 클라이언트 변경 요청 도착 | SENT (그대로) | (아직 없음) |
| 2차 견적 작성 시작 | **SUPERSEDED** | DRAFT |
| 2차 발송 | SUPERSEDED | SENT |
| 클라이언트 합의 | SUPERSEDED | ACCEPTED |

---

## 7.3 마감재 CHANGED 상태 활용

마감재 탭에서 변경된 항목은 **상태를 CHANGED로 전환**해두세요. 이렇게 하면:

- 화면에서 시각적으로 강조 (주황색 등)
- 2차 견적 만들 때 차이 자동 인식
- 변경 이력(MaterialHistory)에 자동 기록

### 권장 워크플로우

```
1. 클라이언트로부터 변경 요청 도착 (예: 욕실 타일 색 변경)
2. 마감재 탭으로 이동
3. 해당 행 펼치기
4. itemName, unitPrice 등 수정
5. 상태를 CONFIRMED → CHANGED로 변경
6. 자동 저장됨 (1초 뒤)
7. 다음 변경 항목으로 이동
```

> 💡 **여러 항목 변경 시**: 모든 변경을 마감재 탭에서 다 마친 후, 그 다음에 견적 탭으로 가서 한 번에 2차 견적을 만드시면 효율적입니다.

---

## 7.4 직전 차수와 비교 (AI 자동 요약)

### 무엇인가

Suplex가 1차 견적과 2차 견적을 비교해서 **변경 사항을 한국어로 자동 요약**해 footerNotes에 넣어주는 기능.

### 사용법

2차 견적 화면에서 **"직전 차수와 비교"** 버튼 클릭. 끝.

### 자동 추가되는 footerNotes 예시

```
※ 본 견적은 2차(변경) 견적입니다. 1차 대비 변경 사항:

  - 강마루 색상 변경: 메이플 → 오크 (전체)
  - 욕실 타일 색상 변경: 화이트 무광 → 그레이 무광 (공용·안방)
  - 거실등 펜던트 추가 (320,000원)
  - 합계 차이: +536,250원

※ 1차 견적(2026-05-10 발행)은 SUPERSEDED 처리되었으며, 본 2차 견적이 유효합니다.
※ 본 2차 견적의 유효기간은 발행일로부터 30일입니다.
```

### 클라이언트 입장에서의 가치

PDF만 받아도 어떤 변경이 어떤 단가로 반영됐는지 한눈에 보입니다. 이게 없으면:
- "왜 53만 원 늘었나요?" → 전화 응대 필요
- "이거 합의 안 한 것 같은데?" → 분쟁

이 작은 자동 요약 한 줄이 위 두 시나리오를 모두 사전 차단합니다.

---

## 7.5 MaterialHistory — 마감재 변경 이력

마감재의 모든 변경은 자동 기록됩니다.

### 기록 내용

| 필드 | 예시 |
|---|---|
| materialId | (어느 마감재) |
| changedById | 김미영 사원 |
| changedAt | 2026-05-13 14:30:25 |
| field | itemName |
| oldValue | "벽 타일 — 600×300 화이트 무광" |
| newValue | "벽 타일 — 600×300 그레이 무광" |

### 어떻게 보나

| 위치 | 방법 |
|---|---|
| 마감재 (고급) 탭 | 행 클릭 → 변경 이력 패널 펼치기 |
| 프로젝트 햄버거 메뉴 | "변동 로그" 클릭 → 전체 프로젝트 변경 이력 |
| JSON 백업 | history 필드에 모든 이력 포함 |

## 7.6 발주 후 마감재 변경 ⚠️ 알림

**가장 위험한 시나리오**입니다:

```
1차 견적 → 합의 → 발주 진행
       ↓
[그런데 클라이언트가 변경 요청]
       ↓
마감재 변경 → ⚠️ 거래처에 이미 발주된 자재
       ↓
거래처에 변경 통보 또는 재발주 필요
```

Suplex는 이 상황을 자동 감지합니다.

### 자동 감지 메커니즘

`PurchaseOrder` 모델에 `materialChangedAt` 필드가 있어, 발주 후 그 마감재가 수정되면 타임스탬프가 자동 기록됩니다. 화면에 ⚠️ 표시.

### 처리 절차

1. 홈 화면 또는 프로젝트 카드의 ⚠️ 알림 클릭
2. 어느 발주의 어떤 마감재가 바뀌었는지 확인
3. 거래처에 즉시 연락 (전화 또는 카카오톡)
4. 옵션 두 가지:
   - **변경 가능 (배송 전)**: 거래처에 변경 통보 → 새 사양으로
   - **변경 불가 (배송됨/생산됨)**: 기존 자재 사용 또는 추가 발주 + 비용 정산

> 💡 발주 후 변경은 비용 발생합니다. 거래처에 따라 위약금·취소 수수료 있으니 발주서에 미리 변경 정책 명시 권장.

## 7.7 분쟁 시 증빙으로 활용

매뉴얼에서 가장 중요한 부분 중 하나입니다.

### 시나리오: "이 가격 합의 안 했다"

공사 막바지에 클라이언트가 "처음 제시한 가격이랑 다르다"고 주장.

### Suplex로 즉시 대응

1. **프로젝트 → 견적 탭**: 모든 견적 차수가 시간순 보존되어 있음
   - 1차 견적 (SUPERSEDED, 2026-05-10 SENT)
   - 2차 견적 (ACCEPTED, 2026-05-14 SENT)
2. **각 견적의 PDF**가 footerNotes에 변경 요약을 포함하고 있음
3. **MaterialHistory**: 어느 항목이 언제 누구에 의해 어떤 단가로 바뀌었는지 모두 기록
4. **카카오톡으로 발송한 PDF 파일**: 외부 메신저에서도 발송 시간·내용 확인 가능

### 분쟁 예방 팁

매뉴얼이 권장하는 운영 습관:

| 시점 | 해야 할 일 |
|---|---|
| 견적 발송 시 | 카카오톡 본문에 "X차 견적서 보내드립니다 (발행일 YYYY-MM-DD)" 명시 |
| 합의 시 | 클라이언트로부터 "확인했습니다 / 합의합니다" 한 마디라도 카카오톡으로 받기 |
| 변경 시 | 클라이언트의 변경 요청 메시지를 캡처해서 메모 탭에 저장 |
| 추가 비용 발생 시 | 발생 즉시 견적에 반영하고 새 차수로 발송. 공사 끝나고 한꺼번에 청구 ❌ |

자세한 시나리오는 [Playbook #8: 분쟁 증빙](playbooks/08-dispute-evidence.md) 참고.

## 7.8 자주 막히는 곳

**Q. 변경이 자잘하게 10번 일어나면 견적이 10차까지 가나요?**
A. 자잘한 변경 여러 개를 **일주일 단위로 묶어서** 한 차수로 발행하시는 게 효율적입니다. 단, 큰 변경은 즉시 차수 갱신.

**Q. 1차 견적을 SUPERSEDED 처리했는데 클라이언트가 "1차로 갈게요"라고 하면?**
A. 그러면 새로 **3차 견적**을 만드세요 (1차와 같은 내용으로). SUPERSEDED는 영구 잠금이고, 1차 자체를 다시 살리지 않습니다. 3차 = "1차로 회귀하기로 결정한 견적".

**Q. 견적 상태 자동 전환은 안 되나요?**
A. **권장 안 합니다.** 자동이면 분쟁 시 모호함이 생깁니다. 수동 전환은 명확한 의사결정 시점이 됩니다.

**Q. 2차 견적 만들 때 1차 복제가 더 빠른가요, 아니면 새로 만드는 게 좋나요?**
A. **변경이 30% 이하면 복제 후 수정**, **30% 이상이면 새로 만들기**가 일반적입니다. 복제는 옵션이 그대로 옴 — 디자인비·부가세 등 다시 입력 안 해도 되는 장점.

**Q. 클라이언트가 PDF를 못 받았다고 하면?**
A. Suplex 안에서 PDF는 매번 새로 생성 가능합니다. 견적 탭에서 해당 차수 클릭 → PDF 인쇄. 카카오톡 발송 이력은 외부 메신저에서 확인.
